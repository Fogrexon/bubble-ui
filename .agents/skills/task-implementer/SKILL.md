# task-implementer

新規機能の実装からPull Request作成までの手順を定義するワークフロー。各フェーズを順を追って実行すること。

## ルールとガイドライン
- 各フェーズは前のフェーズが完了してから確実に次へ進む
- テスト設計・実装（フェーズ5）は、機能実装（フェーズ6）の前に必ず実施する
- 実装（フェーズ6）はテストをパスするまで継続する
- 指定された別スキル（`test-impl`, `self-review`, `pull-request-manager`）も併せて利用する
- **ドキュメント管理**: 作業中ユーザーに読ませるドキュメントは、`llm-docs/xxxx-[ドメイン名]/yyyy-[ドキュメント名]` の形式で4桁の連番を付与して管理する
- **作業中断と引継ぎ**: ユーザーから中断指示があった場合は、`handover-generator` スキルを呼び出して現在の進行状況をまとめた引継ぎドキュメントを作成し、作業を一時停止する

## 手順

1. **要求の理解と初期調査**
   - 要求された内容がない場合は、どのような作業が必要なのかヒアリングする
   - 要求された機能の内容を分析する　必要な場合は追加のフォローアップ質問を行う
   - `git worktree` を用いて、現在の変更から隔離された独立の作業用ディレクトリを作成し、そこへ移動する
   - 関連コードの簡易的な調査を行う
2. **要求と実装内容の定義**
   - 初期調査を基に、どのような機能をどう実装するかをまとめる
   - 疑問点や不明点があればユーザーに確認する
3. **issue登録**
   - フェーズ2の内容をIssueとして登録する
   - **※ Issue登録の手順は `issue-manager` スキルを呼び出して実行する**
4. **詳細調査**
   - 簡易調査の結果をさらに分解し、複数の調査タスクを定義・実施する
   - （例：参考コードの場所、密連携する既存コードの依存関係の把握など）
5. **実装検討、テストのデザイン**
   - 既存機能から新規機能を呼び出すインターフェースを設計する
   - 要求を満たすかを検証するテストをデザインし、先に実装する
   - **※ テストデザイン方針は `test-impl` スキルを呼び出して設計に従う**
5.5. **実装設計**
   - どのスクリプトにどのようなコードを追加するか具体的に分解し、タスクリストを作成する
6. **実装**
   - テストを書いたのち、パスするまで実装を進める
   - 実装中のエラー等でテストの誤りが明らかになった場合のみ、テストやインターフェースを修正してよい
   - 適宜コードフォーマッタを実行する
7. **セルフレビュー**
   - 実装内容のレビューを実施する
   - **※ レビューの観点・手順は `self-review` スキルを呼び出して実行する**
8. **pull requset作成**
   - 実装・テストパス・フォーマット・レビュー完了後、PRを作成する
   - **※ PR作成の手順は `pull-request-manager` スキルを呼び出して実行する**
9. **ユーザー確認とクリーンアップ**
   - 実装およびPR作成完了後、ユーザーに報告しレビューを依頼する
   - ユーザーの確認完了後、使用した `git worktree` の作業ディレクトリをクリーンアップ（削除）する

## 期待される出力
- 登録されたIssue
- 作成・通過済みのテストコード
- フォーマット・レビュー済みの実装コード
- 作成されたPull Request
